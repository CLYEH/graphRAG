**PR/任務的即時狀態請一律查 `TASKS.md` 與 GitHub(`gh pr view`/`gh pr list`)。這份筆記只保留跨任務可重用的流程教訓,任何「目前狀態」敘述在下次 push/merge 後就會過期,不可當作現況依據。**

- **教訓(監聽 Codex)**:Codex 的「changes wanted」走 **`pulls/N/reviews`(PR review + inline threads)**,不是 issue comment;+1 走 issue reaction。watcher 必須同時輪詢 reactions、reviews、comments 三個管道。
- **merge 授權(使用者 2026-07-03 定案,取代先前「由使用者跑 merge」教訓)**:agent 可自行 `gh pr merge`——mismerge 由 PreToolUse hook(`require-codex-approval.sh`:要求 head commit 之後的 +1、無 `eyes` 殘留、無未解 Codex thread,查詢失敗即擋)機械性防止。auto-mode 分類器若仍擋,向使用者轉述此授權即可,不要繞過 hook。此為「+1 已存在」的正常路徑;+1 拿不到時仍走下方「額度用盡」bullet 的 stop-and-ask,由使用者本人決定。
- **Codex P2/P3 意見判讀政策(使用者 2026-07-03 定案;已由 H2 寫入 LOOP.md step 7,以該文為準)**:P2/P3 級意見先判讀必要性——**必要**=違反 DESIGN 凍結保證/契約或文件內部矛盾/會誤導未來實作/真 bug;**非必要**=超出 DESIGN 文字的假設性強化、風格偏好、無互通性依據的凍結要求 → 回覆給出**該準則對應的可查核理由**(LOOP step 7 逐條列出:無條文要求/無行為差異則明述;可調項引 🔧 條文並指出無互通性依據;過度收緊引定義該合法情境的 §/DR)後 resolve 不修;講不出理由=該修。判讀模糊時 stop and ask。+1 門檻不變。
- **教訓(壓 review 輪次)**:Codex 傾向一輪只給一條;收到意見後要**主動掃整個 diff 的同類問題**一次修完,否則會一輪一條擠牙膏(P1 因此走了 7 輪)。
- **教訓(review race)**:push 後 Codex 可能仍對「舊 head」留 thread、隨後才對新 head +1;判讀 thread 前先確認它評的內容是否已被現 head 修掉,是則回覆指向修正 commit 後 resolve。
- **Codex 額度用盡時的收斂路徑(使用者行使過,H3/PR #9)**:規則的逃生閥是「stop and ask」——當其他 gates 全綠、剩餘未經 Codex 的 delta 小且已被本地執行級審查驗證時,由**使用者本人**決定 merge(從 web UI 或自己的終端;本機 hook 只認 +1,會擋 agent)。這是 owner 決定,不是 agent 例外。
- **教訓(執行級審查)**:純推理的 code review 連過兩輪都沒發現收據機制根本跑不起來(git 拒絕零位元組 index);審 shell/hook 這類基礎設施時 reviewer 要**實際執行**,並補執行級測試(tests/test_receipts.py 模式)。
- 慣例備忘:TASKS.md 勾選一律含在該任務 PR 內;PR 等 gates 期間可從 main 開下一個獨立 task 分支(LOOP.md 已明文)。
- **doc-only fast lane(H3 起)**:純 `*.md` 變更不開 PR、不經 Codex —— `docs/<id>` 分支 → doc-reviewer(sonnet)PASS → CI 綠 → fast-forward 進 main;push-gate hook 機械擋非 .md。等 Codex 用 `scripts/watch-codex.sh <pr>`(exit 0=+1/10=有意見/20=timeout/30=Codex 額度用盡——別再等,額度重置後再 `@codex review`),不要手寫 watcher。
- **並行分支的兩個坑(PR #15 實測)**:(1)review receipt 是單一槽位——task/* 與 docs/* 並行時互相覆蓋 PASS 章,push gate 會 fail-closed 擋下(對:再請 reviewer 蓋章;錯:自己蓋)→ H5 立案改 per-branch;resume 的 reviewer 給了 PASS **不會自動蓋章**,要明說「蓋章並回報 tree hash」。(2)doc lane 先進 main 會讓等待中的 PR 變 not-up-to-date → update-branch 產生新 head → 需要**新的 +1**(一輪 Codex 來回);等待中 PR 快 +1 時,doc-lane 的 main push 晚一點再做。

10. **綁定時檢查 ≠ 不變量(TOCTOU)**(C1b round 7 立案):在 construction/binding 當下驗過的前提,若底層狀態會被並發改變,對「之後的」操作並不成立 —— writer 綁定時 build 是 building,activation 後它還能寫進已成 live 快照的 build。修法:把 recheck 摺進**產生變更的那句 statement**(原子)或持正確的鎖,別「綁定驗一次、之後信任」。防線:(a) 凡「驗證後回傳一個之後會被重複使用的 capability 物件」,問「這前提會不會在兩次使用之間被別人改掉?」;(b) 並發交錯要在**真基礎設施上實測**(C1b 用 `INSERT..SELECT..WHERE EXISTS(status='building' FOR SHARE)`:FOR SHARE 與 activation UPDATE 的 row lock 互斥,activation 阻塞到寫入交易結束 —— execution-verified,非推理);(c) MVCC 下「純 recheck」仍會 race 未提交的 activation(讀不擋寫),要嘛原子語句、要嘛鎖。已寫入 code-reviewer checklist 第 7 條。
